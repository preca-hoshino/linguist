## 零阶段：立项与签出 (Phase 0: Initialization & Checkout)
- 接收任何任务后，必须首先将本地 `develop` 分支更新至最新状态，并基于此签出独立的工作分支，**不得基于其他任何非 `develop` 分支派生新分支**。
- 工作分支名称须与任务类型严格对应，命名规范以 [CONTRIBUTING.md](../CONTRIBUTING.md) 为唯一依据。

## 第一阶段：迭代研发与原子提交 (Phase 1: Iterative Dev & Atomic Commits)
- 每完成对**任意单个文件**的变更后，须立即在本地执行：
  ```bash
  npm run check
  ```
- 检查结果须 **100% 通过**（零 Lint 警告、零类型错误、零测试失败）。若未通过，须持续修复直至全部通过，方可进入下一步。
- 验证通过后，须立即以**原子粒度**执行 `git commit`，提交信息格式严格遵循 [CONTRIBUTING.md](../CONTRIBUTING.md) 的消息约定。**禁止将多个文件的变更合并至同一次提交**。
- 本阶段按"变更 → 验证 → 提交"循环推进，响应用户的持续迭代需求。
- **【禁止项】** 在用户未明确下达推送指令（如"可以推送"、"提 PR"等）之前，禁止执行 `git push` 或发起任何 Pull Request。

## 第二阶段：自治监控与提交闭环 (Phase 2: Autonomous PR & Review Loop)
- 收到用户的推送授权指令后，将本阶段积累的所有原子提交推送至远端，并创建 Pull Request。**PR 标题须同样遵循提交信息规范。**
- **【禁止项】** PR 创建后，禁止主动宣告任务完成。须持续轮询（设定合理的间隔与超时机制）等待以下结果：远端 CI 流水线（如 GitHub Actions）执行完毕。
- **评审意见处理规范**：
  - 若收到的意见被判定为**与原始需求相悖、将破坏既有功能，或属于无效要求**，须**禁止自动修改代码**，转而向用户提交一份结构化报告，内容包含：评审意见原文摘要、采纳该意见后预计产生的负面影响分析，由用户作出最终决策。
  - 若收到的意见被判定为**合理且符合项目规范**，须自动执行修复，完成后重新执行 `npm run check` 确保全部通过，再依次执行原子提交与 `git push`，使 PR 恢复至通过状态。
- 以上监听与修复循环持续运行，直至 PR 状态变更为 **Merged（已合并）**，方可视为本次任务完结。
