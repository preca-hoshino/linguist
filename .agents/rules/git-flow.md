## 1. 分支策略 (Branching Strategy)
- 提出任何任务要求前，必须首先参考 [CONTRIBUTING.md](../CONTRIBUTING.md) 中的分支命名规范。
- **必须从 `develop` 分支签出新分支**。
- 确保所选分支名称与即将进行的修改类型（功能、修复、重构等）严格匹配。

## 2. 验证与修复规范 (Verification & Fix)
- 对**任何一个文件**（包括新增、修改、删除）的操作完成后，必须立即执行以下命令：
  ```bash
  npm run check
  ```
- **必须确保所有自动化检查均以 100% 通过**（无 Lint 警告、类型错误或测试失败）。
- 如果检查未通过，必须修复至完全通过为止，方可进行下一步。

## 3. 原子化提交 (Atomic Commits)
- **每次验证通过一个文件后，必须立即进行原子化提交**。
- 提交信息规范（包括 Type 类型、格式等）请**严格参考 [CONTRIBUTING.md](../CONTRIBUTING.md) 中关于提交消息的最新约定**。
- 不得将多个验证后的文件堆积在一次提交中（保持小步快跑，多次提交）。

## 4. 合并规范 (Merging)
- 代码通过所有本地验证后，需在获得用户明确同意的前提下，才能根据 [CONTRIBUTING.md](../CONTRIBUTING.md) 的标准流程（建议通过 PR 流程）合并入目标分支。
- **发起 PR 请求时，PR 标题也必须严格遵循提交规范。**
